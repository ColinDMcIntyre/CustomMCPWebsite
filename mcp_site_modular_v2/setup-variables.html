Setup & Variables

Set it up without getting buried in variables

This guide is organized in the order most people actually need: defaults first, generated secrets second, your account-based values after that, and the rest at the end. Most of the work is copy and paste.

What you are setting up

  • A personal MCP endpoint mounted at /mcp
  • Static MCP OAuth support for clients that expect OAuth
  • Optional static bearer token fallback for simple testing
  • Google OAuth refresh-token mode for personal use
  • GitHub variables for the included GitHub tooling

Before you start

  1. Create or log into GitHub.
  2. Create or log into Render.
  3. Have your Google OAuth client ready or be prepared to create it.
  4. Have a GitHub personal access token if you want the GitHub tools working right away.

How the auth options work

The server supports two MCP auth paths at the same time. Most people should use static OAuth and can also generate a bearer token for manual testing or scripts if they want that path ready. You do not need to fill every value on day one.

Recommended: static OAuth

Use this for ChatGPT Developer Mode or any MCP client that expects an OAuth flow. This build uses a static client ID and secret, not dynamic client registration.

Bearer token path

Use this if you want a simple token for manual testing, scripts, or a backup access path.

How the static OAuth setup works

This server is set up around static MCP OAuth as the main recommendation. That keeps the path clearer for ChatGPT Developer Mode and similar MCP clients, while still letting you keep a token fallback if you want one.

Use static OAuth when

  • You want cleaner MCP client support.
  • You are connecting through ChatGPT Developer Mode.
  • You want a normal OAuth-style path with refresh tokens.

Use token fallback when

  • You want a simpler manual test path.
  • You are hitting the endpoint yourself from a script.
  • You want a backup auth path in case you need it later.
